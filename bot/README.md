# Telegram-бот для заявок на товары

Python-бот на aiogram 3 для оформления заявок на заказ товаров. Заявки хранятся в PostgreSQL, бот работает через polling.

## Стек

- Python 3.11+
- aiogram 3
- PostgreSQL
- SQLAlchemy async
- asyncpg
- pydantic-settings
- Railway

## Структура

```text
src/
  main.py
  config.py
  database.py
  bot/
  handlers/
  keyboards/
  services/
  repositories/
  models/
  states/
```

## Локальный запуск

Создайте и активируйте виртуальное окружение:

```bash
python -m venv .venv
source .venv/bin/activate
```

Для Windows PowerShell:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

Установите зависимости:

```bash
pip install -r requirements.txt
```

Создайте `.env` по примеру `.env.example`:

```env
APP_ENV=dev
BOT_TOKEN=token_from_botfather
ADMIN_CHAT_ID=-1001234567890
ADMIN_IDS=123456789,987654321
DATABASE_URL=postgresql://user:password@localhost:5432/poizon_bot
ADMIN_USERNAME=admin_username
```

`APP_ENV=dev` включает безопасные локальные заглушки, например тестовую оплату прямо в Telegram. Для Railway/боевого запуска ставьте `APP_ENV=production`.

`ADMIN_CHAT_ID` — ID чата, куда бот отправляет заявки. `ADMIN_IDS` — Telegram ID админов через запятую. Админские команды и кнопки работают только в `ADMIN_CHAT_ID`, а если `ADMIN_IDS` заполнен, то ещё и только для этих пользователей.

Для тестов вне Railway `DATABASE_URL` можно не указывать. Тогда бот попробует подключиться к локальному PostgreSQL:

```text
postgresql://postgres:postgres@127.0.0.1:5432/poizon_bot
```

В этом случае локально должна существовать база `poizon_bot`, пользователь `postgres` с паролем `postgres`, и PostgreSQL должен быть запущен.

Самый простой вариант для локального теста — поднять PostgreSQL через Docker из папки `bot`:

```bash
docker compose up -d db
```

После этого запускайте бота:

```bash
python -m src.main
```

Запустите бота из папки `bot`:

```bash
python -m src.main
```

Таблицы создаются автоматически при старте.

## Логи

Логи пишутся в консоль через стандартный Python `logging`.

Локально вы видите их прямо в терминале, где запущена команда:

```bash
python -m src.main
```

На Railway откройте проект, выберите worker-процесс бота и перейдите во вкладку `Logs`. Там будут сообщения о запуске, ошибках, создании заявок, действиях админа и dev-оплате.

## Railway

`Procfile` — это файл-инструкция для Railway/Heroku-подобных платформ. Он говорит платформе, какой процесс нужно поднять после деплоя.

В этом проекте `Procfile` содержит:

```text
worker: python -m src.main
```

`worker` означает фоновый процесс без HTTP-сервера. Это подходит для Telegram-бота на polling, потому что он сам постоянно опрашивает Telegram API.

1. Создайте PostgreSQL service в Railway.
2. Добавьте переменные окружения:
   - `BOT_TOKEN`
   - `APP_ENV=production`
   - `ADMIN_CHAT_ID`
   - `ADMIN_IDS`
   - `DATABASE_URL`
   - `ADMIN_USERNAME`
3. Убедитесь, что Railway запускает проект из папки `bot`.
4. Procfile уже содержит команду:

```text
worker: python -m src.main
```

## Сценарий работы

Пользователь нажимает `Оформить заявку`, бот создаёт черновик со статусом `draft` и показывает меню заявки. Поля редактируются через FSM и сразу сохраняются в PostgreSQL.

После ввода поля бот не отправляет новую карточку заявки, а редактирует исходное inline-меню заявки.

Пользователь может открыть свои заявки кнопкой `Мои заявки` или командой:

```text
/orders
```

После отправки бот проверяет обязательные поля: адрес, тип товара, размер, ссылка или фото. Если всё заполнено, заявка получает статус `sent_to_admin` и отправляется в админ-чат.

Администратор может одобрить заявку, указать цену, отклонить заявку с причиной или связаться с пользователем. После одобрения пользователь получает цену и кнопки оплаты, отказа и связи с админом.

В админ-чате доступна команда статистики:

```text
/stats
```

Команды `/start`, `/orders` и `/stats` регистрируются при старте бота, поэтому Telegram показывает их при нажатии на значок `/`.

## Оплата

Сейчас оплата реализована как заглушка в `src/services/payment_service.py`: бот отдаёт тестовую ссылку и не создаёт реальный платёж. Для официальной работы на территории России лучше подключать российского платёжного провайдера, например ЮKassa. У ЮKassa есть API для создания платежей, redirect-сценарий с `confirmation_url`, webhooks для подтверждения результата и отдельный блок по отправке чеков по требованиям 54-ФЗ.

Ниже общий план подключения. Это технический план, а не юридическая или налоговая консультация: перед запуском продаж всё равно нужно уточнить схему у бухгалтера или платёжного провайдера.

Что нужно подготовить:

- Официальный статус продавца: ИП, ООО или самозанятый, если ваша модель продаж подходит под НПД.
- Расчётный счёт или способ получения выплат, который поддерживает выбранный провайдер.
- Договор с платёжным провайдером, например с ЮKassa.
- Понимание по чекам: по 54-ФЗ обычно нужна онлайн-касса и фискализация чеков; для самозанятых действует отдельная логика чеков по 422-ФЗ. У ЮKassa есть документация по обоим вариантам.
- Публичные документы магазина: правила оплаты, возврата, контакты продавца, политика обработки персональных данных.
- HTTPS backend для webhook-уведомлений от платёжного провайдера. Один Telegram polling-бот не заменяет webhook оплаты, потому что провайдер должен отправить серверу подтверждение, что деньги реально списаны.

Какие переменные добавить в `.env`:

```env
PAYMENT_PROVIDER=yookassa
PAYMENT_SHOP_ID=
PAYMENT_SECRET_KEY=
PAYMENT_RETURN_URL=https://t.me/your_bot
PAYMENT_WEBHOOK_SECRET=
```

Что означает каждая переменная:

- `PAYMENT_PROVIDER` — какой провайдер используется. Сейчас можно оставить `yookassa`.
- `PAYMENT_SHOP_ID` — ID магазина в кабинете провайдера.
- `PAYMENT_SECRET_KEY` — секретный ключ API. Его нельзя публиковать в GitHub и нельзя отправлять пользователю.
- `PAYMENT_RETURN_URL` — куда вернуть пользователя после страницы оплаты. Обычно это ссылка на бота или отдельная страница “Спасибо за оплату”.
- `PAYMENT_WEBHOOK_SECRET` — секрет для дополнительной проверки webhook, если вы добавите свою подпись или защитный токен в URL.

Как должен работать официальный поток оплаты:

1. Админ одобряет заявку и указывает цену.
2. Бот вызывает `PaymentService`.
3. `PaymentService` создаёт платёж через API провайдера: сумма в RUB, описание, ID заявки, `return_url`, данные чека.
4. Провайдер возвращает объект платежа и ссылку подтверждения оплаты, например `confirmation.confirmation_url`.
5. Бот сохраняет `payment_id` и `confirmation_url` в базе. Сейчас в модели есть `payment_url`; позже лучше добавить отдельное поле `payment_id`.
6. Пользователь нажимает `Оплатить` и переходит на официальную страницу провайдера.
7. Провайдер проводит оплату и отправляет webhook на ваш backend.
8. Backend проверяет, что событие пришло от провайдера, находит заявку по `payment_id` или metadata `order_id`, затем меняет статус заявки на `paid`.
9. Бот уведомляет пользователя и админов, что заявка оплачена.

Для ЮKassa технически нужно будет отправлять запрос создания платежа примерно с такими данными:

```json
{
  "amount": {
    "value": "1500.00",
    "currency": "RUB"
  },
  "confirmation": {
    "type": "redirect",
    "return_url": "https://t.me/your_bot"
  },
  "capture": true,
  "description": "Оплата заявки #123",
  "metadata": {
    "order_id": "123"
  }
}
```

Если вы обязаны отправлять фискальные чеки через провайдера, к платежу добавляется объект `receipt`: данные покупателя, позиции заказа, количество, цена, ставка НДС и признак способа/предмета расчёта. Конкретные значения зависят от вашей налоговой схемы и товара, поэтому их нужно согласовать с бухгалтером.

Почему нужен webhook:

Пользователь может закрыть страницу оплаты, потерять интернет или не вернуться в Telegram. Поэтому нельзя считать оплату успешной только потому, что пользователь нажал кнопку. Надёжный источник правды — уведомление от платёжного провайдера.

Что нужно добавить в код позже:

- HTTP-приложение, например FastAPI, для webhook:

```text
POST /payments/webhook
```

- Метод в `PaymentService`, который создаёт платёж через API провайдера.
- Поля в модели заявки: `payment_id`, `payment_status`, возможно `paid_at`.
- Метод проверки webhook. В идеале дополнительно запрашивать платёж по `payment_id` через API провайдера и убеждаться, что статус действительно `succeeded`.
- Метод в `OrderService`, который переводит заявку в статус `paid`.
- Уведомление пользователю и админам после оплаты.

На Railway обычно запускают два процесса:

```text
worker: python -m src.main
web: uvicorn src.web.main:app --host 0.0.0.0 --port $PORT
```

`worker` — Telegram polling-бот. `web` — HTTP-сервер для webhook оплаты. На первом этапе можно оставить только `worker` и тестовую ссылку в `payment_service.py`.
